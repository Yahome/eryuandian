# TWA-001 前置架构说明汇总

## 结论

TWA-001 **不能直接进入 App Shell UI 实现**。必须先冻结技术栈、数据库方向、部署基准、Mock / Adapter 边界、100 用户同时在线 MVP 基准，以及移动端优先设计基线。确认前，`yuan-frontend` 不得开工。

## 推荐技术栈

### 前端

- React
- Vite
- TypeScript
- React Router
- TanStack Query
- CSS 方向优先 Tailwind CSS；如沿用现有 CSS，也必须明确移动端优先规则

理由：适合 Web App MVP，类型安全、路由清晰、mock / adapter 切换成本低。

### 后端

- Node.js 单体应用
- Hono 优先，Fastify 备选
- API 层以 contract / mock / adapter 为主

理由：MVP 先上线、稳定、可维护，不拆微服务。

### 数据库

- PostgreSQL
- ORM 方向优先 Drizzle ORM，Prisma 备选

理由：登录、会话、额度流水、生成任务、作品记录、账户记录都是典型关系型数据。

### 部署

- Docker Compose
- 当前飞牛 OS / VM103 作为基准
- 反代用 Caddy 或 Nginx
- PostgreSQL 单实例部署

理由：足够覆盖 MVP，避免 K8s / 分布式数据库 / 复杂微服务过早介入。

### 静态资源 / 资产存储

- mock 阶段用静态 fixture 和本地占位
- 真实对象存储留给后续任务

### 任务队列

- mock 阶段不强制引入
- 真实模型接入前再评估 Redis / 队列 / worker

## 数据库选择

推荐 PostgreSQL 作为 MVP 上线数据库。

- 不建议一开始使用复杂分布式数据库
- SQLite 只适合本地开发 / demo，不建议作为真实上线主库
- PostgreSQL 足够支撑 100 用户同时在线的 MVP 基准
- 当前默认禁止创建真实 schema、migration、ORM model

## 100 用户同时在线 MVP 基准

当前目标是稳定服务约 100 个同时在线用户，不是大规模商业化架构。

mock 阶段应支持：
- 页面访问
- 登录 mock
- 额度展示 mock
- 生成任务状态 mock
- 作品记录 mock
- loading / error / empty 基础态

真实模型接入前需要补：
- 队列
- 重试
- 限流
- 幂等
- 额度预占 / 回滚
- 对象存储
- 审计与监控

## 移动端优先设计基线

- 移动端是主设计，PC 是增强
- 目标宽度至少覆盖 375px / 390px / 430px
- TWA-001 必须从移动端开始设计 App Shell
- 底部 Tab 导航优先
- 顶部轻量 header
- 单列内容流
- 触控区域建议不小于 44px
- 避免 hover-only
- 表单适合手机输入
- 考虑微信 / 手机浏览器 viewport
- 考虑 safe-area
- PC 端在移动端基础上扩展为双栏或多栏
- TWA-007 只做移动端体验回归，不承担第一次移动端适配

## App Shell / 首页 / 工作台边界

### 允许

- `/`
- `/app`
- `/app/dashboard`
- 基础 App Shell
- 首页
- 工作台
- 移动端底部导航
- PC 增强导航
- 空状态
- loading / error 基础态
- 静态 mock 数据

### 禁止

- 真实登录
- 真实短信
- 真实生图
- 真实试卷生成
- 真实支付
- 真实额度扣费
- 真实数据库 schema
- 真实 API handler
- secret / token / key

## 推荐后续任务顺序

推荐采用 B+：

1. TWA-001：技术栈冻结 + 移动优先 App Shell / 首页 / 工作台基础
2. TWA-002：登录与会话 Mock / Adapter
3. TWA-003：额度账本 Mock / Adapter
4. TWA-004：营销生图工作台 Mock 流程
5. TWA-005：试卷生成工作台 Mock 流程
6. TWA-006：我的作品 / 生成记录
7. TWA-007：账户 / 商业化占位 + 移动端体验回归

ROADMAP 建议等待用户确认后再做全量同步，不在本轮偷偷改完。

## yuan-frontend 放行条件

必须同时满足：

- 用户确认技术栈
- 用户确认数据库选择
- 用户确认移动端优先原则
- 用户确认 App Shell 范围
- 用户确认后续任务顺序
- yuan-reviewer 对架构说明无 blocker

## 风险与注意事项

- 技术栈未冻结就写 UI 会返工
- 移动端后置会导致大规模重构
- 额度账本后置会导致生图 / 试卷流程返工
- Mock / Adapter 边界不清会误接真实 provider
- 过度设计会拖慢上线
- 过早创建数据库 schema 会增加迁移成本
- dashboard / ROADMAP / TASK_BOARD 事实源漂移会制造错觉

## 验收标准

- 技术栈、数据库、ORM、部署、100 用户同时在线基准、移动端优先原则说明清楚
- App Shell 允许 / 禁止边界清楚
- B+ 后续任务顺序清楚
- frontend 放行条件清楚
- 风险与边界清楚
- 不实现 UI、不创建真实 API、不修改数据库 schema、不接真实短信 / 模型 / 支付
