## yuan-architect 意见

### 总体判断

`yuan-architect` 判断：用户新版本 B 更合理，但建议形成“B+ 收敛版”。核心观点是顺序按 B，范围吸收当前 A 的 App Shell 完整性，并将移动端基础能力前移，而不是完全丢掉移动端适配。

推荐主线：

1. TWA-001：App Shell + 首页 / 工作台 / 路由 / 导航 / 空状态 / 响应式基础
2. TWA-002：登录与会话 Mock / Auth Contract / Adapter
3. TWA-003：额度账本 Mock / Adapter
4. TWA-004：生图工作台 Mock 流程
5. TWA-005：试卷工作台 Mock 流程
6. TWA-006：我的作品 / 生成记录
7. TWA-007：账户与商业化占位

### 当前版本 vs 用户新版本对比

`yuan-architect` 认为 A 的优点是与当前 TWA-000、ROADMAP、README、summary 一致，且先做 App Shell 和登录没有问题；但 A 的主要问题是额度放到 TWA-006 太晚，生图和试卷都需要额度展示、消费、失败回滚等语义，后补会造成硬编码和返工。A 的 TWA-007 作为移动端适配偏晚，且账户页缺位，与 TWA-000 已冻结的 `/app/account` 范围不一致。

`yuan-architect` 认为 B 的优势是依赖顺序更清楚：Shell → Auth → Quota → 业务工作台 → 聚合作品 → 账户商业化占位。B 更利于 backend contract 分层和真实 provider 替换边界。但 B 的 TWA-001 名称偏瘦，如果只做空路由，不做首页 / 工作台 / 导航 / 空状态 / 响应式基础，会比 A 退步；同时移动端适配不能消失。

### 每个任务的前置依赖与交付边界

- TWA-001：依赖 TWA-000。应交付公开首页、App Shell、`/app` 到 `/app/dashboard` 默认入口、一级导航、页面空状态、基础响应式规则；不做真实登录、真实 API、真实额度、生图、试卷、支付。
- TWA-002：依赖 TWA-001。应交付手机号登录 mock、固定验证码策略、mock session、退出登录 mock、Auth adapter 接口；禁止真实短信、真实 token 存储、真实 LoginCode 落库、真实 session 表。
- TWA-003：依赖 TWA-002 的用户上下文。应交付 `QuotaSummaryDTO`、`CreditLedgerDTO[]`、额度卡片、账本预览、消费模拟函数 / adapter；禁止真实订单、会员、支付、数据库 schema。
- TWA-004：依赖 Shell、登录态、额度语义。应交付生图表单、mock 任务、mock 状态、mock 资产和 `costCredits` 展示；禁止真实 AI 生图模型、provider key、真实对象存储和真实扣费。
- TWA-005：依赖 Shell、登录态、额度语义和 Exam contract。应交付试卷表单、mock 任务、mock 试卷预览、答案解析；禁止真实试卷模型、真实导出、真实题库 schema。
- TWA-006：依赖生图资产和试卷产物。应交付聚合列表、筛选、最近生成记录、空状态和分页占位；禁止真实对象存储、真实删除、真实下载计费。
- TWA-007：依赖登录、额度和账本摘要。应交付账户页、用户资料、手机号脱敏、额度摘要、账本摘要、付费入口占位、套餐占位；禁止支付、订单、会员表、真实套餐购买。

### 对关键争议的判断

- 是否提前做额度账本：是。`yuan-architect` 明确区分 Phase 1 的额度 contract + mock/adapter 与 Phase 3 的真实额度系统，认为 TWA-003 前置不等于商业化越界。
- 是否保留 TWA-007 移动端适配：不建议保留为单独兜底任务；建议移动端基础前移到 TWA-001，并进入后续每个任务验收。
- 是否改成账户与商业化占位：建议改，因为 TWA-000 已冻结 `/app/account`。
- 是否建议新增 TWA-008：`yuan-architect` 没有强制要求新增 TWA-008，但建议如果采纳 B+，需要一次事实源同步 / 决策记录。
- TWA-001 是否应该包含首页和工作台：必须包含，但应是 MVP 壳和空状态，不是完整精装修。

### 可能风险

`yuan-architect` 建议后续写入 RISKS 或任务文档的风险：额度账本提前后被误解为真实商业化系统；TWA-001 范围膨胀；移动端适配被取消后后续页面不可用；TWA-004 / TWA-005 越界接真实模型；TWA-006 变成资产系统；TWA-007 越界到支付；ROADMAP、TWA-000、README、summary、agent 文档之间的事实源漂移。

### 建议是否改动 ROADMAP

`yuan-architect` 建议改，但不是本轮改。应先由 `yuan-control` 取得确认，再由 `yuan-architect` 做事实源同步任务。建议记录一条决策：Phase 1 将额度账本 mock/adapter 前置到业务工作台之前；移动端基础纳入 App Shell 与各任务验收；TWA-007 改为账户与商业化占位。边界是不实现真实额度系统、支付、会员、订单、真实 provider、数据库 schema。

## yuan-frontend 意见

### 总体判断

`yuan-frontend` 判断：B 的依赖顺序更合理，但不能原样照搬，建议采用“B 为主 + 吸收 A 的首页工作台 + 移动端基础前移”的混合修正方案。

主要结论：

- 额度账本提前到 TWA-003 是对的，至少应做 Adapter + UI 消费链路。
- 不建议把 TWA-007 完全替换成账户与商业化占位；移动端第一轮适配必须保留，否则 TWA-001 至 TWA-006 很可能返工。
- App Shell 不应只做路由壳，应至少包含首页/工作台框架、导航、空状态、loading/error 基础态、响应式基础栅格。
- A 的最大问题是额度能力后置和移动端后置。

### 推荐前端开发顺序

`yuan-frontend` 推荐：

1. TWA-001：App Shell + 首页/工作台框架（路由骨架、导航、空状态、基础响应式）
2. TWA-002：登录与会话 Mock/Adapter（auth guard、session 生命周期）
3. TWA-003：额度账本 Mock/Adapter（quota summary + ledger preview + 预扣/扣减/回滚前端状态）
4. TWA-004：营销生图工作台（依赖 001/002/003）
5. TWA-005：试卷工作台（依赖 001/002/003）
6. TWA-006：我的作品/生成记录（依赖 004/005 的任务与产物模型）
7. TWA-007：账户与商业化占位 + 移动端第一轮适配；如果不新增 TWA-008，建议 007 不要二选一。

### App Shell 边界

`yuan-frontend` 建议 App Shell 包含：`/`、`/app/*`、`/dev-dashboard` 隔离边界；侧栏/顶部导航、当前路由高亮、可简化面包屑；loading / empty / error / unauthorized；未登录重定向策略占位；断点体系、容器宽度、栅格、卡片密度规则；设计令牌接入。App Shell 不应包含生图/试卷具体业务表单、额度消费实际规则、商业化支付流程。

### 各任务前端难点

- TWA-001：壳层过薄会后续返工，过厚会侵入业务。
- TWA-002：session 状态与路由守卫一致性、mock 登录态刷新恢复。
- TWA-003：额度状态要和生成任务状态联动，否则后续展示会假。
- TWA-004：长表单、异步任务状态机、结果列表一致性。
- TWA-005：应与生图共享任务态和结果态抽象，避免重复造轮子。
- TWA-006：跨来源聚合 image/exam，需要统一筛选、排序、分页、空态。
- TWA-007：账户页信息架构与商业化占位文案边界，移动端改造不能破坏桌面端密度。

### 可复用组件建议

`yuan-frontend` 建议从一开始抽取：AppLayout / AppHeader / AppSidebar、RouteGuard / AuthGate、AsyncStatePanel、QuotaBadge / QuotaCard / LedgerPreviewList、JobStatusBadge、AssetCard、FilterBar + CursorPagination、SectionCard / FieldRow / OptionChips。

### Mock 数据建议

`yuan-frontend` 建议 mock 层集中管理，不散落组件：`mock-auth`、`mock-quota`、`mock-generation`、`mock-exam`、`mock-assets`。组件只消费 DTO，并应预置失败样本，例如额度不足、任务失败、超时。

### 移动端适配建议

`yuan-frontend` 不建议移动端最后一次性做完，建议两层：TWA-001 落基础响应式（断点、栅格、导航折叠、卡片流式布局）；TWA-007 做页面级精修或回归。如果 TWA-007 改成账户与商业化占位，则 TWA-001 至 TWA-006 里必须显式写“基础适配完成”。

### 前端越界风险

`yuan-frontend` 标出的风险：mock 规则硬编码进 UI 组件；前端擅自扩展 API 字段 / 状态枚举；前端实现真实扣费/支付语义；私接真实短信 / 生图 / 试卷 / 支付入口；账户与商业化占位误做成真实订单流；过晚做响应式导致 CSS 大返工。

### 可能涉及的前端文件或模块

只列可能路径，不在本轮修改：`src/app/router/*`、`src/app/layout/*`、`src/modules/auth/*`、`src/modules/quota/*`、`src/modules/image-workbench/*`、`src/modules/exam-workbench/*`、`src/modules/assets/*`、`src/modules/account/*`、`src/shared/components/*`、`src/shared/adapters/*`、`src/styles/*`。

## yuan-backend 意见

### 总体判断

`yuan-backend` 判断：从后端 / Mock / Adapter 可落地性看，B 比 A 更合理，但建议 B+ 轻微修正：先把登录会话和额度账本前置，再开生图/试卷；移动端适配不应占用 TWA-007，TWA-007 更适合“账户与商业化占位 + 账本可解释性闭环”。

`yuan-backend` 认为 A 更偏前端体验驱动，后端 contract 闭环偏晚；B 更符合登录 → 额度账本 → 生图/试卷的真实依赖。

### 推荐后端 / Mock / Adapter 顺序

`yuan-backend` 推荐：

1. TWA-001 App Shell 路由骨架（含基础空状态和响应式基础，不做移动端精修）
2. TWA-002 登录与会话 Mock/Adapter（send-code/login/me contract）
3. TWA-003 额度账本 Mock/Adapter（quota summary + ledger preview + reserve/debit/release 语义）
4. TWA-004 生图工作台（严格走 quota adapter，先 mock provider）
5. TWA-005 试卷工作台（同上）
6. TWA-006 我的作品（聚合 assets + exam paper，消费结果可追溯到账本 ref）
7. TWA-007 账户与商业化占位（会员/支付入口占位、账本明细入口、风控提示文案位）

`yuan-backend` 备注：移动端第一轮精修建议单独新建后续任务，不占 TWA-007。

### 额度账本是否前置

`yuan-backend` 明确建议前置到 TWA-003。理由是生图/试卷都依赖可消费、可预占、可回滚的额度语义；若额度后置，TWA-004 / TWA-005 只能假消耗，后续要改状态机与错误码。TWA-003 应提前统一不足额、预占失败/释放失败、任务失败回滚策略；MVP 阶段只 contract 化和 mock，不上真实事务。

### API contract 依赖

`yuan-backend` 给出的依赖链：

- TWA-002 提供 `/api/auth/send-code`、`/api/auth/login`、`/api/me`。
- TWA-003 提供 `/api/quota` + ledger preview，以及内部 quota adapter：reserve/debit/release。
- TWA-004 / TWA-005 消费上述能力：创建任务时 reserve，成功 debit，失败 release。
- TWA-006 读取 assets/exam 聚合，并展示 job/ledger 的可追溯关系。
- TWA-007 只做账户商业化入口 contract 占位，不接真实支付。

`yuan-backend` 额外发现一个风险点：`API_CONTRACT.md` 旧草案与 TWA-000 contract 命名有漂移，例如 `/api/auth/sms-code` vs `/api/auth/send-code`、`/api/generations/images` vs `/api/generation/image`。建议进入 RISKS 或对应任务文档“待对齐项”。

### Mock provider 建议

`yuan-backend` 建议从 TWA-002 起统一可替换 provider 接口：auth provider、quota provider、image provider、exam provider、asset repository。Mock 输出应可复现，前端只消费 DTO，不感知 provider 类型。

### 未来真实服务替换点

替换点包括：短信网关 adapter、会话存储 adapter、quota ledger adapter、image model adapter、exam model adapter、asset storage adapter、payment/billing adapter。当前都只保留 mock / adapter 边界，不落真实服务。

### 当前阶段是否需要真实数据库 schema

`yuan-backend` 结论：不需要，而且应明确暂不需要。当前只应冻结 contract 与 mock 行为。

### 后端越界风险

`yuan-backend` 标出的风险：TWA-002 / TWA-003 提前落真实 DB schema；TWA-004 / TWA-005 直连真实模型；TWA-007 偷跑支付接口；前端展示字段被硬编码成业务事实；TWA-000 与 API_CONTRACT.md 的 contract 漂移不收口。

### 推荐任务调整

`yuan-backend` 建议采用 B 或 B+；将移动端适配第一轮从 TWA-007 移出，单列后续任务；TWA-003 明确产出额度账本状态机、错误码草案、reserve/debit/release contract；TWA-004 / TWA-005 明确禁止真实模型接入；TWA-001 验收标准明确 App Shell 包含首页入口、app 路由骨架、导航、空状态、响应式基础；增加一条“contract 漂移清单”子任务用于统一 API 命名。

## yuan-reviewer 意见

### 总体判断

`yuan-reviewer` 判断：建议采用 B 的方向，但不要原样照抄；推荐“B + 移动端基础内置”的顺序。本轮只读完成，未写代码、未改文件、未提交、未创建 TWA-001、未改 ROADMAP / dashboard / schema / API / DB / secret。

`yuan-reviewer` 认为 A 能跑 UI，但业务依赖倒挂；B 更合理，但移动端不能完全消失，至少应作为每个 UI 任务的响应式基础验收内置。

### 验收风险

`yuan-reviewer` 标出两个 Blocker 级风险：

1. 若采用 B 但不更新事实源，会产生 closeout 漂移。当前 ROADMAP、TWA-000、web-app README、dashboard summary 仍是 A 顺序。如果后续直接按 B 创建 TWA-001，Dashboard / ROADMAP / task docs 会打架。
2. App Shell 如果包含首页、工作台、路由、导航、空状态、响应式基础，TWA-001 容易过大。应限制为极简入口、Shell、路由占位、导航、空状态、基础响应式，不做完整业务页面。

Major 风险：额度账本晚于生图/试卷导致高返工；账户页在 TWA-000 范围内但 A 没有对应任务；真实 API 文件边界需要写清楚，防止 contract 与真实实现混淆。

### 任务拆分风险

- TWA-001：应做路由占位、导航、登录态占位、空状态、基础响应式；不做完整生图表单、试卷表单、作品列表、真实登录、额度扣减。
- TWA-002：必须定义 session 状态、未登录态、登录态、退出、过期 mock；固定验证码仅 mock；禁止真实短信、真实 token secret、真实落库。
- TWA-003：必须是 Mock/Adapter，不是真实商业化系统；只做 QuotaSummary、CreditLedger fixture、消费/预占/释放 mock 行为；不做订单、支付、会员、发票、真实 DB schema。
- TWA-004 / TWA-005：依赖登录态 + 额度；可以前端先 mock，但必须依赖统一 DTO、job status、quota adapter；禁止真实模型和 provider key。
- TWA-006：应晚于生图和试卷，聚合 `GeneratedAssetDTO | ExamPaperDTO`；不应发明第二套资产模型。
- TWA-007：账户与商业化占位合理，但必须限制为资料、手机号脱敏、额度摘要、账本摘要、套餐入口 Coming soon；禁止支付、订单、套餐真实价格体系、checkout、支付 SDK。

### closeout 漂移风险

`yuan-reviewer` 指出如果改成 B，必须由 `yuan-control` / `yuan-architect` 统一同步事实源，不能只改一个地方。需同步的文件包括 ROADMAP、TWA-000、web-app README、summary、TASK_BOARD、RISKS、CHANGELOG、agent docs、dashboard JSON 等。但本轮不执行修改。

`yuan-reviewer` 还提醒：TWA-000 是已完成冻结任务，直接改 TWA-000 内容会改变历史事实源。更稳做法是新建任务顺序调整 / 架构补充决策文档，或由 `yuan-control` 记录一次后续决策，再同步事实源。

### 必须增加的验收标准

通用验收建议：

- 边界验收：无真实短信、真实模型、真实支付、secret、数据库 schema、dashboard schema 改动；`/dev-dashboard` 不混入 Web App 业务运行数据。
- Mock / Adapter 验收：mock 集中管理，frontend 只依赖 DTO / adapter，失败态/loading/empty/unauthenticated 全覆盖。
- 响应式验收：桌面和移动宽度都要验证，无横向滚动，导航可用，表单/卡片/列表不溢出。
- 依赖验收：生图 / 试卷提交必须经过统一 quota adapter；我的作品只能聚合既有 DTO；账户页只能读取 me / quota / ledger mock。
